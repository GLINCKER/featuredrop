# Core Architecture

FeatureDrop flips the script on traditional product adoption tools. Instead of injecting a slow, third-party JavaScript bundle into your `window` object that polls an external server, **FeatureDrop is compiled directly into your application bundle as a set of pure React components and state machines.**

You own the rendering, the storage, and the analytics.

## The Data Flow

The architecture is highly deterministic. UI components never fetch their own data; they strictly subscribe to a unified React Context provided by `<FeatureDropProvider>`.

1. Manifest source is passed into `FeatureDropProvider`.
2. Storage adapter provides seen/dismissed state to `FeatureDropProvider`.
3. UI components (`NewBadge`, `ChangelogWidget`, `Tour`) consume provider state.
4. Component interactions emit analytics callbacks.

### Understanding the Engine (Context)

When you mount `<FeatureDropProvider />`, it kicks off a specific reconciliation loop:

<Steps>
### Read Storage
The provider queries your configured `StorageAdapter` to retrieve an array of Feature IDs this specific user has already seen or clicked.

### Evaluate Targeting
The provider takes the arrays of `seen` IDs and maps them against the provided `Manifest`. It evaluates audience targeting arrays, semver versions, and expiration dates (`hideAfter`).

### Broadcast State
It broadcasts the resulting set of "Active / Unseen" features down to the React Context. The `<NewBadge>` components instantly react to this context and appear active.
</Steps>

<Callout type="warning">
**Performance Note:** Because FeatureDrop validates all rules strictly in memory on the client-side, the reconciliation loop takes **less than 1ms**. The bottleneck is strictly limited by how fast your custom `StorageAdapter` can return the user's history payload. 
</Callout>

### Headless By Default

While `<ChangelogWidget />` and `<Tour />` come with beautiful default CSS, they are internally designed using the **Headless UI render-prop pattern**.

If you pass a function as the `children` to any FeatureDrop component, we strip our CSS entirely and yield the internal state directly to you:

```tsx filename="MyCustomChangelog.tsx"
<ChangelogWidget>
  {({ isOpen, toggle, features, count }) => (
    <div className="my-custom-company-ui">
       <button onClick={toggle}>
          Updates <span className="badge">{count}</span>
       </button>
       
       {isOpen && (
         <ul>
           {features.map(f => <li key={f.id}>{f.title}</li>)}
         </ul>
       )}
    </div>
  )}
</ChangelogWidget>
```

This ensures you are never fighting global CSS overrides just to match your brand's unique design system.
